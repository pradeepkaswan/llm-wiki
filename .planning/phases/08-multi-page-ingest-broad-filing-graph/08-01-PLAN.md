```typescript
export interface RippleTarget {
  slug: string;

  updatedSlugs: string[];
  skippedSlugs: string[];
}

export async function rippleUpdates(
  primaryArticles: Article[],
  store: WikiStore,
  schema: string,
): Promise<RippleResult>
```

Implementation per locked decisions:
1. Call `store.listArticles()` to get all existing articles
2. Build MiniSearch index via `buildIndex(existingArticles)`
3. For each primary article, query BM25 with `title + ' ' + summary`
4. Collect top 10 results per primary, exclude all primary article slugs (D-03)
5. Filter results below BM25 score threshold of 3.0 (same as `BM25_DEDUP_THRESHOLD`)
6. Deduplicate target slugs across all primaries (a target may match multiple primaries — only update once)
7. If no targets remain, return empty result
8. Make a single LLM call (D-04) with: primary article titles/summaries, target article titles/summaries, and the schema. Prompt instructs LLM to return JSON array of `{slug, seeAlsoText}`. Use `generateText()` with `temperature: 0.2, maxOutputTokens: 2048`.
9. Parse LLM output: strip markdown code fences (`/^\`\`\`json?\n?/` and `/\n?\`\`\`$/`), then `JSON.parse()`. On failure, log warning to stderr and return empty result (per RESEARCH Pitfall 5).
10. For each target in parsed result, load target article via `store.getArticle(slug)`, call `upsertSeeAlsoEntry(article.body, seeAlsoText)`, save via `store.saveArticle(updatedArticle, 'update')`. Process sequentially (not Promise.all) per RESEARCH anti-pattern on parallel writes.
11. Log entries auto-generated by saveArticle's appendLog (D-05). Add "ripple from [primary-slug]" in the description field.

Import `buildIndex` and `search` from `../search/search-index.js`. Import `generateText` from `../llm/adapter.js`. Import `upsertSeeAlsoEntry` from `./see-also.js`.

**Create `src/synthesis/backlink-enforcer.ts` (per D-12 through D-16)**

```typescript
import type { Article } from '../types/article.js';
import type { WikiStore } from '../store/wiki-store.js';
import { upsertSeeAlsoEntry } from './see-also.js';

const WIKILINK_RE = /\[\[([^\]|]+)(?:\|[^\]]+)?\]\]/g;

export async function enforceBacklinks(
  article: Article,
  store: WikiStore,
): Promise<string[]>  // returns slugs that were updated
```

Implementation:
1. Extract all `[[wikilinks]]` from `article.body` using WIKILINK_RE regex. Handle both `[[slug]]` and `[[slug|display text]]` — capture group 1 gives the slug.
2. Deduplicate extracted slugs into a `Set<string>`.
3. Remove the article's own slug from the set (prevent self-referential backlinks).
4. For each target slug (sequentially, per RESEARCH anti-pattern on parallel writes):
   a. `store.getArticle(targetSlug)` — if null, skip (target doesn't exist)
   b. Build backlink entry: `[[${article.slug}]] - Related: ${article.frontmatter.title}`
   c. Call `upsertSeeAlsoEntry(targetArticle.body, backlinkEntry)` — returns updated body
   d. If body unchanged (upsertSeeAlsoEntry returned same string), skip save (idempotency — avoids unnecessary index rebuild)
   e. If body changed, create updated article object preserving all frontmatter, save via `store.saveArticle(updatedTarget, 'update')`
5. Return array of slugs that were actually saved.

Key constraints:
- Per D-14: All reads through `store.getArticle()`, all writes through `store.saveArticle(article, 'update')`. Never write to disk directly.
- Per D-15: Idempotent — `upsertSeeAlsoEntry` handles duplicate detection.
- Per D-16: This is complementary to the wikilink sanitizer, not merged. The sanitizer strips invalid forward links; this adds missing backward links.
- Per RESEARCH Pitfall 4: This function is NOT recursive. It runs one level only. The caller (ask.ts, file.ts) decides when to invoke it — never from inside WikiStore or ripple.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM output -> JSON.parse | Ripple module receives free-form text from LLM, parses as JSON |
| Article body -> wikilink extraction | Regex extraction from user-authored markdown content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Tampering | ripple.ts JSON.parse | mitigate | Strip markdown code fences before JSON.parse; wrap in try/catch; on failure return empty result (no throw). Test with malformed JSON input. |
| T-08-02 | Tampering | backlink-enforcer.ts slug extraction | mitigate | WIKILINK_RE regex captures only slug portion (no path separators); all file writes go through WikiStore.saveArticle which uses slugify() — path traversal impossible |
| T-08-03 | Denial of Service | ripple.ts unbounded targets | mitigate | BM25 top-10 cap per D-03 limits ripple to at most 10 article updates per primary article; sequential processing prevents runaway parallel writes |
| T-08-04 | Denial of Service | backlink-enforcer.ts recursive loop | mitigate | enforceBacklinks() is not recursive — only called from command-level pipeline (ask.ts, file.ts), never from inside WikiStore or ripple per-target save loop |
</threat_model>
