<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| schema.md content -> LLM prompt | Schema read from disk is injected into user prompts sent to LLM providers |
| LLM article output -> schema co-evolution | Categories from LLM-generated articles are appended to schema taxonomy |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-05 | Spoofing | schema.md content injection into prompts | accept | Schema is a local file in user's vault. No external actor can modify it. User has full control. The content is plain Markdown text injected as user-prompt context, not executable code. |
| T-07-06 | Tampering | LLM categories appended to schema without validation | mitigate | Categories come from LLM output but are filtered through extractSchemaCategories case-insensitive dedup before append. No arbitrary content injection — only category names (short strings) are appended in a controlled format ("- **Name**: Auto-added"). |
| T-07-07 | Repudiation | Schema changes not tracked | mitigate | Every updateSchema() call triggers appendLog('schema', ...) per D-07. The activity log provides a chronological audit trail of all schema mutations. |
| T-07-08 | Information Disclosure | Schema content exposes wiki structure to LLM provider | accept | Schema contains category names and formatting conventions — no sensitive personal data. This is the intended behavior per SCHEMA-02 (LLM must read schema). User chose to send this to their configured LLM provider. |
</threat_model>
